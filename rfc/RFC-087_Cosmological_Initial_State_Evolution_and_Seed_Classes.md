## 1. Executive Summary

This RFC defines what we can and cannot claim cosmologically when we repeatedly apply the COG update rule from different initial seeds.

Key result:
1. The currently locked minimal kernels evolve node states on a fixed DAG.
2. They do not add nodes or edges.
3. Therefore, "cosmological expansion" is not generated by update alone in these kernels.
4. Expansion, horizon growth, and large-scale geometry evolution require an explicit topology-growth (spawn) contract.

This RFC turns that distinction into a testable program with seed classes, expected signatures, and falsification criteria.

---

## 2. Problem Statement

Question addressed:
1. If we start from a singularity-like or dense initial condition (or one vacuum node) and repeatedly apply the update rule, what do we see?

Current risk:
1. Mixing state-update conclusions with topology-growth conclusions.
2. Overclaiming cosmology from fixed-topology runs.

Policy in this RFC:
1. Separate state dynamics from topology dynamics.
2. Require explicit lane labeling in all cosmology artifacts.

## 3. Two Cosmology Lanes

## 3.1 Lane A: Fixed-Topology Cosmology (available now)

Implemented by:
1. `minimal_world_kernel.py`
2. `minimal_world_kernel_projective_unity.py`

What evolves:
1. node states `psi(v,t)`.

What does not evolve:
1. node set `V`,
2. edge set `E`.

Interpretation:
1. This lane studies phase/state equilibration, attractors, and boundedness on a fixed causal scaffold.
2. It is not a full universe-growth model.

## 3.2 Lane B: Dynamic-Topology Cosmology (blocked by spawn closure)

Required for:
1. expansion-like behavior,
2. horizon growth,
3. large-scale structure emergence,
4. singularity-to-expansion narrative at graph level.

Status:
1. blocked until spawn/edge-addition contract is locked and replay-validated.

## 4. Seed Class Contract

Cosmology runs must declare one of the following seed classes:

1. `S0_single_vacuum_node`
   - one node, no parents, vacuum-aligned state.

2. `S1_single_dense_node`
   - one node, no parents, high-amplitude or high-load coefficient state.

3. `S2_dense_clique_patch`
   - finite high-connectivity patch (near-complete parent overlap), nontrivial state load.

4. `S3_sparse_random_patch`
   - finite sparse patch used as control against dense starts.

All seeds must include:
1. exact JSON microstate,
2. deterministic hash,
3. declared kernel profile and policy IDs.

## 5. What We Expect to See (by Lane and Seed)

## 5.1 Lane A (fixed topology): expected signatures

### S0: single vacuum node
1. no parent messages implies identity payload each tick,
2. state remains invariant under the minimal update rule,
3. no expansion because topology is fixed.

### S1: single dense node
1. same topological behavior as S0 (no new nodes),
2. state is static under no-parent identity payload,
3. no blow-up from repeated updates in isolation.

### S2: dense clique patch
1. strong mixing from heavy parent overlap,
2. rapid transient dynamics in node states,
3. integer profile may show coefficient growth in state values,
4. projective-unity profile remains bounded by projector closure,
5. topology still fixed; no geometric expansion claim allowed.

### S3: sparse random patch
1. slower mixing relative to S2,
2. lower coupling intensity and weaker transient gradients,
3. useful baseline for dense-start comparisons.

## 5.2 Lane B (dynamic topology): expected signatures (programmatic, not yet closed)

If spawn is later locked:
1. S0 should produce outward causal growth from the root,
2. S1/S2 should initially show local density saturation then outward relaxation fronts,
3. effective horizon/diameter trajectories become meaningful observables,
4. expansion rate and structure growth become policy-dependent and must be preregistered.

## 6. Cosmology Observable Contract

Every run must emit:
1. `N_nodes(t)` and `N_edges(t)` (Lane A: constant; Lane B: dynamic),
2. edge density `rho(t) = 2E / (N(N-1))`,
3. diameter and depth summaries,
4. state-load metrics (max coefficient norm, mean norm, entropy proxies),
5. attractor/cycle diagnostics,
6. profile metadata (`kernel_profile`, projector policy if used).

Derived indicators:
1. `expansion_indicator = dN_nodes/dt` (only meaningful in Lane B),
2. `saturation_indicator` for dense-region relaxation,
3. `anisotropy_indicator` across directional partitions (for isotropy checks).

## 7. Falsification Conditions

This RFC is falsified (or must be revised) if any of the following occur:

1. Lane labeling ambiguity:
   - an artifact claims expansion-like conclusions from fixed-topology runs without explicit topology-growth mechanism.

2. Determinism failure:
   - identical seed + profile + steps produce different outputs.

3. Contract mismatch:
   - cosmology conclusions are reported without required observables and profile metadata.

4. Scope inflation:
   - singularity/Big-Bang physical claims are promoted without dynamic-topology evidence lane.

---

## 8. Cosmological Implications (Current, Defensible)

From current locked kernels and contracts:
1. A finite seed remains finite at every finite tick.
2. No literal divergence appears inside fixed finite topology.
3. "Single node universe expands by update alone" is not supported by current minimal kernels.
4. Dense starts show state-dynamic transients, not automatic geometric expansion.
5. Singularity-like language should map to finite high-density initial conditions (RFC-066 framing), not mathematical infinities.

---

## 9. Governance Locks

1. Cosmology artifacts must include `cosmo_lane` (`fixed_topology` or `dynamic_topology`).
2. No claim may mix integer and unity/projective profiles in one closure record.
3. Cold/warm start controls from RFC-079 apply to cosmology runs.
4. Any topology-growth claim requires explicit spawn policy ID and replay proof.
